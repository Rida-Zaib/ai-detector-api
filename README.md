# ai-detector-api
Content Authenticity Examiner

I built this to solve a gap I ran into while working on an auto-grading assignments platform: the grader could handle typed and image-based answers just fine, but had no way to tell whether a student's answer was actually written by them or generated by an AI. So instead of bolting that logic onto the existing app, I built it as its own standalone API — something that can be dropped into any grading tool, LMS, or form, not just the one I originally built it for.

What it actually does


Guesses whether text is AI-generated, using a mix of writing-pattern statistics and an LLM that reasons about the actual content, not just surface style
Admits when it's not sure. If a piece of writing could plausibly be either a skilled human or an AI, it says so instead of pretending to be confident. Good grammar isn't treated as proof of anything — plenty of humans write cleanly too
Refuses to score gibberish. Random keystrokes or test strings get rejected before any AI/human label is even attempted, because that label only means something for actual language
Checks for plagiarism against everything previously submitted to the system
Grades answers on whether they're actually correct — not on grammar, not on AI-likelihood. A messy but correct answer should score well; a polished but wrong one shouldn't
Tracks typed vs. pasted input on the frontend, as an extra signal for academic integrity review
Reads text out of .docx, .pdf, and images, not just plain text boxes


How it's built

Nothing fancy, just clean separation of concerns:


TextExtractor and Detector are abstract base classes — they define what a thing should do, not how
Concrete extractors (DocxExtractor, PdfExtractor, ImageExtractor) and detectors (PatternDetector, SemanticDetector) implement that contract however makes sense for them
CompositeDetector doesn't care whether it's combining a cheap local heuristic or a remote LLM call — it treats every detector the same way
SubmissionRepository is the only thing that touches SQL directly; everything else just calls .save() and moves on


Stack

FastAPI backend, Groq for the LLM calls (meta-llama/llama-4-scout-17b-16e-instruct), python-docx and pdfplumber for document parsing, SQLite for storage, and a plain HTML/JS frontend with no framework — so it's easy to drop into anything.

Running it locally

bashpip install -r requirements.txt

Set your Groq key as an environment variable (grab one free at console.groq.com):

bash# Windows PowerShell
$env:GROQ_API_KEY = "your_key_here"

# macOS/Linux
export GROQ_API_KEY="your_key_here"

Then:

bashuvicorn main:app --reload --port 8001

Open static/showcase.html to use it, or hit http://127.0.0.1:8001/docs for the interactive API docs.

API

POST /detect/report — send either text or a file (.docx, .pdf, .png, .jpg), plus optional paste_percentage and question fields.

bashcurl -X POST http://127.0.0.1:8001/detect/report \
  -F "text=Your submission text here" \
  -F "question=What causes seasons on Earth?"
